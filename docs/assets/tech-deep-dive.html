Med-Recallix 技术原理深度解析

从 SM-2 算法到 AI Agent 记忆系统 — 核心技术实现揭秘

面向开发者的核心脉络

  • 复习引擎：SuperMemo-2（SM-2）间隔重复 — 医学记忆曲线的可执行实现。
  • Agent：三层记忆（画像 / 长期 / 当日）+ 上下文流水线 + 会话压缩。
  • 平台：EdgeOne Pages KV 作为唯一持久化、Next.js API 与 Edge KV 的代理链、Web Crypto + JWT。

一、SM-2 间隔重复算法

SM-2 是 Med-Recallix 与通用闪卡应用的核心差异点：复习日期不是随意设定，而是由易度因子（EF）与历史表现共同决定。

1.1 背景

SuperMemo-2 由 Piotr Wozniak 于 1987 年提出，是间隔重复家族中实现简单、工业界广泛采用的一支。项目通过 npm 包 supermemo 直接调用标准 SM-2 更新规则。

1.2 核心公式（易度因子 EF）

每次评分后更新易度因子（q 为 0–5 的等级）：

EF' = EF + (0.1 - (5 - q) * (0.08 + (5 - q) * 0.02))

据此再推导间隔天数与重复次数 repetition。实现中同时维护三个关键量：interval（距下次复习的天数）、repetition（连续成功轮次）、efactor（易度因子，初值通常为 2.5）。

1.3 医学场景下的 0–5 分

  • 0–2：回忆失败或严重困难 —— 需缩短间隔、降低 EF，强化再认。
  • 3：勉强想起 —— 进入巩固区，间隔适中。
  • 4–5：流畅回忆 —— 延长间隔，推向长期记忆。

1.4 代码实现（与仓库一致）

import { supermemo } from "supermemo";

export function calculateNextReview(card: Card, grade: ReviewGrade): Card {
  const result = supermemo(
    { interval: card.interval, repetition: card.repetition, efactor: card.efactor },
    grade,
  );
  const nextDue = new Date();
  nextDue.setDate(nextDue.getDate() + result.interval);
  return {
    ...card,
    interval: result.interval,
    repetition: result.repetition,
    efactor: result.efactor,
    dueDate: toISODateString(nextDue),
    lastReviewDate: toISODateString(new Date()),
  };
}

二、AI Agent 记忆系统

三层记忆 + 流水线

2.1 三层架构

  • Layer 1 — UserProfile（长期、结构化）：考试目标、强弱项、偏好等，持久化为用户画像。
  • Layer 2 — LongTermMemory（情景、有界）：从对话中抽取的条目，最多约 100 条 episodic 记忆，支持检索与淘汰。
  • Layer 3 — DailyEpisode（当日、易失）：按自然日聚合的学习活动（时长、复习数、主题等），用于「今天学了什么」的上下文。
flowchart LR M[用户消息] --> K[提取关键词] K --> R[召回相关记忆] R --> P[加载 UserProfile] P --> E[加载 DailyEpisode] E --> S[拼装 System Prompt] S --> L[LLM 生成回复]

2.2 记忆写入与会话压缩

  • 记忆抽取：在 AI 主回复之后，可触发第二次 LLM 调用，从本轮对话抽取可存储条目并写入 LongTermMemory（具体策略见实现模块）。
  • 会话压缩：当消息条数超过阈值（实现中为 COMPACTION_KEEP_RECENT = 20）时，较早消息经摘要合并进会话的 summary 字段，保留尾部消息与摘要并存。

三、KV 存储架构

无传统关系型数据库：业务与配置均以 JSON 序列化字符串存入 EdgeOne Pages KV，读写通过统一 kvGet / kvPut 抽象。

3.1 命名空间

  • med_data：用户、卡片、知识点、会话等业务数据（绑定变量名 MED_DATA）。
  • med_config：系统级配置如 ai_config（绑定变量名 MED_CONFIG）。

3.2 键命名约定（节选）

模式示例数据

kp_index_{userId}kp_index_abc知识点索引列表
deck_{userId}deck_abcDeck（全部卡片）

chat_idx_{userId}chat_idx_abc会话列表索引
cat_{userId} / streak_{userId}—分类树、连续学习统计等

3.3 适配器链（与 kv.client.ts 一致）

  1. globalThis 绑定：生产环境若存在 MED_CONFIG / MED_DATA Edge KV 绑定，直接走原生 API。
  2. HTTP 代理：生产环境无绑定时，通过 /api/kv/* 转发到 Edge Function 访问 KV。
  3. 本地文件：开发环境默认 ~/.med-recallix/kv/{namespace}/ 下按 key 存 JSON。
  4. 内存 Map：无文件系统时（如部分 Edge 本地场景）退化为进程内 Map，重启即失。
为何 KV 而非 SQL：Serverless 部署、零运维、全球边缘分布、与读多写少的个人学习数据访问模式匹配；复杂查询在应用层以 JSON 结构完成。

四、认证与安全

4.1 密码：PBKDF2（Web Crypto）

  • 算法：PBKDF2-HMAC-SHA256，100,000 次迭代，16 字节随机盐，导出 32 字节哈希（以 hex 存储）。
  • 不用 bcrypt：bcrypt 依赖 Node 原生模块，与 Edge Runtime 不兼容；PBKDF2 为 Web Crypto 标准，浏览器与 Edge 均可运行。

4.2 JWT 与 Cookie

  • 库：jose；算法 HS256；过期时间 7 天（JWT_EXPIRY = "7d"）。
  • Cookie：HttpOnly + SameSite=Lax；部署在 EdgeOne 默认 HTTPS 上，可不强制 Secure（按环境可再收紧）。

4.3 中间件与 userId 解析

  • Middleware：拦截需保护的页面与 /api/*，公开 /api/auth/login、/api/auth/register，其余 API 无有效 JWT 时返回 401。
  • getUserId：优先读取中间件注入的 x-user-id；若缺失则回退为校验 Cookie 内 JWT（解决 EdgeOne 上 header 未传入 Route Handler 的情况）。

五、AI 集成

  • 栈：Vercel AI SDK（ai）+ @ai-sdk/openai-compatible，可对接任意 OpenAI 兼容端点（Moonshot/Kimi、DeepSeek、OpenAI 等）。
  • 流式：streamText 推送 token，前端呈现打字机效果。
  • 配置位置：AI Key / baseURL / model 等存于 KV（如 ai_config），可通过设置页运行时修改，而非仅依赖构建期环境变量。
  • 测验生成：要求模型输出结构化 JSON，服务端用正则等方式解析选项与解析字段，失败时友好报错。

六、EdgeOne 部署架构

flowchart TB GP[Git Push] --> EB[EdgeOne 构建 pnpm install + build] EB --> CDN[静态资源 CDN] EB --> EF[Edge Functions] EF --> KV[(KV 绑定)] API[Next.js API Route Node 上下文] --> PROXY[HTTP fetch /api/kv] PROXY --> EF
KV 代理原因：Next.js 部分 API 运行在 Node 运行时，无法直接持有 Edge 平台的 KV binding；通过 Edge Function 暴露的 /api/kv/* 代理读写，实现「业务代码统一 kvGet/kvPut」。

PWA

Service Worker 对静态资源采用 cache-first、对 API 采用 network-first（具体路由以项目 sw 配置为准），兼顾离线壳与数据新鲜度。

七、前端技术栈

  • 框架：Next.js 16 App Router、React 19、TypeScript strict。
  • 样式：Tailwind CSS v4（PostCSS 管线，无传统 tailwind.config.js 依赖时可由 CSS 入口驱动）。
  • 组件：shadcn/ui（Button、Card、Dialog、Input 等）。
  • 移动端：max-w-lg 单列、底部导航、safe-area 适配刘海屏。
  • 主题：OKLCH CSS 变量，支持暗色模式与系统偏好。
